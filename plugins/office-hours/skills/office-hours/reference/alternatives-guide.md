# 方案生成指南

方案生成是强制的。在确认方向前，必须提供至少 2 个差异化方案。

---

## 方案结构

每个方案必须包含：

```
方案 X: [名称]
  概要: [1-2 句说明核心思路]
  工作量: [S/M/L/XL]
  风险: [低/中/高]
  优点: [2-3 点]
  缺点: [2-3 点]
  复用: [可复用的现有代码/模式]
```

---

## 方案类型

### 1. 最小可行方案

**特征**：
- 最少文件
- 最小 diff
- 最快上线
- 解决核心问题

**适用**：
- 验证假设
- 快速迭代
- 资源有限

### 2. 理想架构方案

**特征**：
- 最佳长期轨迹
- 最优雅设计
- 可扩展性强
- 技术债务少

**适用**：
- 已验证的方向
- 长期投资
- 团队有带宽

### 3. 创意/侧向方案

**特征**：
- 意外路径
- 不同问题框架
- 非常规方法
- 可能更简单

**适用**：
- 常规方案不理想
- 探索可能性
- 创新需求

---

## 生成技巧

### 1. 约束变体

对同一方向施加不同约束：

- 最快上线版本
- 零依赖版本
- 纯前端版本
- 纯后端版本

### 2. 抽象层级变体

从不同抽象层级思考：

- 直接解决问题
- 改变问题框架
- 自动化问题根源
- 让问题不存在

### 3. 技术栈变体

使用不同技术：

- 自建 vs 购买
- 同步 vs 异步
- 本地 vs 云端
- 实时 vs 批处理

## 推荐格式

```
推荐: 选择 [方案 X] 因为 [一句话原因]
```

推荐必须：
- 基于方案对比
- 考虑用户约束
- 有明确理由

---

## 常见错误

### 错误 1: 方案过于相似

**问题**：三个方案只是同一思路的细微变体

**修正**：确保方案有本质差异，不是同一方案的不同实现

### 错误 2: 没有最小可行方案

**问题**：所有方案都是"完整平台"

**修正**：必须有一个方案能在几天内上线

### 错误 3: 偏向推荐

**问题**：描述方案 A 时明显比 B 更正面

**修正**：每个方案的优缺点应该平衡

### 错误 4: 忽略现有代码

**问题**：所有方案都从零开始

**修正**：明确说明每个方案可复用的现有模式

## 示例

```
方案 A: 直接扩展
  概要: 在现有模块上添加新功能
  工作量: S
  风险: 低
  优点:
    - 最快上线（~2天）
    - 复用现有基础设施
    - 学习曲线低
  缺点:
    - 增加模块复杂度
    - 不是长期理想架构
  复用: auth 模块、数据库连接池

方案 B: 独立微服务
  概要: 新建独立服务，通过 API 集成
  工作量: M
  风险: 中
  优点:
    - 独立扩展
    - 清晰边界
    - 技术栈自由
  缺点:
    - 运维复杂度增加
    - 需要服务发现
    - 调试更难
  复用: 共享工具库

方案 C: 第三方集成
  概要: 使用现有 SaaS 解决，自建薄包装层
  工作量: S
  风险: 中
  优点:
    - 开发量最小
    - 成熟方案
    - 免费维护
  缺点:
    - 依赖第三方
    - 定制能力有限
    - 可能有成本
  复用: 无

推荐: 选择方案 A 因为需要快速验证需求，且现有模块已有良好抽象
```
