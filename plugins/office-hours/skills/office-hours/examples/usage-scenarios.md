# 使用场景示例

## 场景 1: 创业想法验证

**用户输入**：
> 我有个想法，想做一个帮助中小企业管理库存的 SaaS

**触发**：用户描述新想法，在写代码前

**流程**：
1. 判断模式 → Startup Mode
2. Q1 需求真实性 → 追问具体证据
3. Q2 现状替代 → 了解用户当前如何管理
4. Q3 目标具体性 → 要求具体到人
5. Q4 最小切入点 → 缩小到最小可付费版本
6. 前提挑战 → 验证假设
7. 方案生成 → 2-3 个方案
8. 输出设计文档

---

## 场景 2: 内部项目规划

**用户输入**：
> 公司想做一个内部工具，自动化我们的报表流程

**触发**：用户描述内部项目

**流程**：
1. 判断模式 → Startup Mode (Intrapreneurship)
2. Q2 现状替代 → 了解当前报表如何生成
3. Q4 最小切入点 → 重构为"最小可 demo 版本"
4. Q6 未来适配 → 重构为"重组后能否存活"
5. 方案生成 → 2-3 个方案
6. 输出设计文档

---

## 场景 3: 副业项目探索

**用户输入**：
> 周末想做个好玩的 side project，帮我 brainstorm 一下

**触发**：用户提到 side project / brainstorm

**流程**：
1. 判断模式 → Builder Mode
2. 最酷版本是什么？
3. 你会展示给谁？
4. 最快可用路径？
5. 方案生成 → 2-3 个方案
6. 输出设计文档

---

## 场景 4: 需求价值评估

**用户输入**：
> 这个需求值得做吗？

**触发**：用户直接问价值评估

**流程**：
1. 判断模式 → 根据项目类型
2. 应用对应模式的问题
3. 前提挑战
4. 方案生成
5. 输出设计文档 + 明确建议

## 输出示例

### 设计文档片段

```markdown
# 设计: 智能库存管理 SaaS

生成时间: 2024-01-15
状态: DRAFT
模式: Startup

## 问题陈述
中小企业缺乏简单、可负担的库存管理工具。

## 需求证据
- 与 3 位店主交流，均表示每月花 4+ 小时手动盘点
- 其中 1 位因库存错误损失 ¥5000/月
- 现有方案要么太贵（¥2000+/月），要么太复杂

## 现状替代
- Excel 手动记录
- 微信群里喊"还有多少 X"
- 每月关门盘点 2 小时

## 目标用户与最小切入点
- 目标: 10-50 人规模的零售店主
- 最小切入点: 仅做库存预警 + 微信通知

## 前提
1. 店主愿意为节省时间付费 — 同意
2. 微信通知是足够的价值点 — 待验证
3. 库存预警比完整管理系统更重要 — 同意

## 方案对比
### 方案 A: 微信小程序
- 最快上线，但功能受限

### 方案 B: Web + 微信通知
- 更灵活，开发量稍大

## 推荐方案
方案 A — 快速验证假设

## 下一步行动
联系 5 位店主，展示 Figma 原型，观察他们最关心哪个功能
```
