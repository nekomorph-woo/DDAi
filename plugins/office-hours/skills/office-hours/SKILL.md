# 需求价值诊断

在编写代码前，深度分析需求/想法的可行性和价值。

**核心原则**：确保问题被理解后，再提出解决方案。输出是设计文档，不是代码。

---

## 流程

### Phase 1: 模式判断

通过 AskUserQuestion 询问用户目标：

| 选项 | 模式 |
|------|------|
| 创业/内部创业项目 | Startup Mode |
| 黑客马拉松/开源/学习/娱乐 | Builder Mode |

### Phase 2A: Startup Mode — 六道强制问题

针对创业/内部项目，逐个追问以下问题（通过 AskUserQuestion，一次一个）：

| 问题 | 目的 | 红旗信号 |
|------|------|----------|
| **Q1: 需求真实性** | 找出真实需求的证据 | "有人感兴趣"、"waitlist" |
| **Q2: 现状替代** | 用户现在怎么解决？ | "目前没有解决方案" |
| **Q3: 目标具体性** | 具体谁最需要？ | "中小企业"、"企业客户" |
| **Q4: 最小切入点** | 最小可付费版本是什么？ | "需要先建完整平台" |
| **Q5: 观察与意外** | 真实观察用户了吗？ | "我们做了调研"、"按预期进行" |
| **Q6: 未来适配** | 3年后产品更不可或缺吗？ | "市场每年增长20%" |

**智能路由**（根据产品阶段跳过部分问题）：
- 无产品阶段 → Q1, Q2, Q3
- 有用户无付费 → Q2, Q4, Q5
- 有付费客户 → Q4, Q5, Q6
- 纯工程/基建 → Q2, Q4

**追问原则**：
- 模糊回答会被追问："你说'企业客户'，能具体到一个公司、一个人吗？"
- 第一次回答通常是修饰版，第二次追问才是真实答案
- 具体性是唯一货币

**快速通道**：若用户提供完整计划或表示不耐烦 → 跳至 Phase 4。

---

### Phase 2B: Builder Mode — 创意协作

针对娱乐/学习/开源项目，通过 AskUserQuestion 逐个询问（生成式，非审问式）：

- **最酷的版本是什么？** 什么让它真正令人兴奋？
- **你会展示给谁？** 什么让他们说"哇"？
- **最快可用的路径是什么？**
- **最接近的现有产品是什么？你的有什么不同？**
- **如果有无限时间你会加什么？** 10x 版本是什么？

**态度**：热情、有观点的协作者，帮助找到最令人兴奋的版本。

**模式切换**：若用户提到客户、收入、融资 → 自动升级到 Startup Mode。

---

### Phase 3: 前提挑战

在提出方案前，挑战前提假设：

1. **这是正确的问题吗？** 不同框架能否产生更简单/更有影响力的方案？
2. **如果什么都不做会怎样？** 真实痛点还是假设？
3. **现有代码是否已部分解决？** 复用现有模式/工具。

输出前提声明，通过 AskUserQuestion 确认：

```
前提：
1. [陈述] — 同意/不同意？
2. [陈述] — 同意/不同意？
3. [陈述] — 同意/不同意？
```

---

### Phase 4: 方案生成（强制）

生成 2-3 个差异化实现方案。这不是可选的。

每个方案包含：

```
方案 A: [名称]
  概要: [1-2 句]
  工作量: [S/M/L/XL]
  风险: [低/中/高]
  优点: [2-3 点]
  缺点: [2-3 点]
  复用: [现有代码/模式]
```

**规则**：
- 至少 2 个方案，非平凡设计建议 3 个
- 一个必须是 **"最小可行"**（最少文件、最小 diff、最快上线）
- 一个必须是 **"理想架构"**（最佳长期轨迹、最优雅）
- 一个可以是 **创意/侧向**（意外路径、不同框架）

**推荐**：选择 [X] 因为 [一句话原因]。

通过 AskUserQuestion 展示，必须获得用户批准后继续。

### Phase 5: 设计文档

将设计文档写入项目 `plans/` 目录。

```bash
mkdir -p plans
DATETIME=$(date +%Y%m%d-%H%M%S)
```

写入 `plans/design-{title}-{datetime}.md`。

#### Startup Mode 模板

```markdown
# 设计: {标题}

生成时间: {date}
状态: DRAFT
模式: Startup

## 问题陈述
{from Phase 2A}

## 需求证据
{from Q1 — 具体引用、数字、行为}

## 现状替代
{from Q2 — 用户当前的具体工作流}

## 目标用户与最小切入点
{from Q3 + Q4}

## 前提
{from Phase 3}

## 方案对比
### 方案 A: {名称}
{from Phase 4}
### 方案 B: {名称}
{from Phase 4}

## 推荐方案
{选择及原因}

## 开放问题
{未解决的问题}

## 成功标准
{可衡量的标准}

## 下一步行动
{一个具体的现实行动 — 不是"去开发"}

## 观察记录
{对用户思维方式的观察性反思，引用具体言论，2-4 点}
```

#### Builder Mode 模板

```markdown
# 设计: {标题}

生成时间: {date}
状态: DRAFT
模式: Builder

## 问题陈述
{from Phase 2B}

## 核心亮点
{让人说"哇"的因素}

## 前提
{from Phase 3}

## 方案对比
### 方案 A: {名称}
{from Phase 4}
### 方案 B: {名称}
{from Phase 4}

## 推荐方案
{选择及原因}

## 开放问题
{未解决的问题}

## 成功标准
{"完成"的定义}

## 下一步
{具体构建任务 — 第一、第二、第三步}

## 观察记录
{对用户思维方式的观察性反思，引用具体言论，2-4 点}
```

通过 AskUserQuestion 确认：
- A) 批准 — 标记 Status: APPROVED
- B) 修订 — 指定需要修改的部分
- C) 重来 — 返回 Phase 2

## 核心原则

### Startup Mode 原则

| 原则 | 说明 |
|------|------|
| **具体性是唯一货币** | "企业客户"不是用户，需要名字、角色、公司、原因 |
| **兴趣 ≠ 需求** | waitlist/signup 不算需求，行为和付费才算 |
| **用户的话胜过创始人pitch** | 用户描述的价值才是真相 |
| **观察，不要演示** | 坐在用户旁边看他们挣扎，不要引导 |
| **现状是真正的竞争对手** | 不是竞品，是用户当前的 spreadsheet + Slack 拼凑方案 |
| **窄胜宽，早期** | 最小可付费版本比完整平台更有价值 |

### Builder Mode 原则

| 原则 | 说明 |
|------|------|
| **兴奋是货币** | 什么让人说"哇"？ |
| **可展示的东西** | 任何东西的最好版本是存在的版本 |
| **解决自己的问题** | 为自己构建，信任那个直觉 |
| **先探索后优化** | 先试奇怪的想法，后打磨 |

## 响应姿态

### Startup Mode

- **直接，不残酷**：目标是清晰，不是拆解。"这是个红旗"比"这值得思考"更有用
- **追问一次，再追问一次**：第一个答案通常是修饰版
- **赞扬具体性**：当创始人给出真正具体、基于证据的回答时，承认它
- **命名常见失败模式**：识别并直接指出

### Builder Mode

- **热情、有观点的协作者**：帮助构建最酷的东西
- **建议他们没想到的酷东西**：相邻想法、意外组合
- **以具体构建步骤结束**：输出是"下一步做什么"，不是"去采访谁"

---

## 重要规则

- **不开始实现**：此技能输出设计文档，不输出代码
- **问题逐个问**：永远不要在一个 AskUserQuestion 中批量多个问题
- **下一步行动是强制的**：每个会话以一个具体现实行动结束
- **完整计划跳过 Phase 2**：若用户提供完整计划，跳过 Phase 2 但仍运行 Phase 3 和 Phase 4
