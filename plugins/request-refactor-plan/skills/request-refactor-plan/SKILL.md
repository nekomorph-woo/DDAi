# 重构计划

通过用户访谈收集需求，生成详细重构计划并创建 issue。

## 平台判断（流程开始时执行）

**ALWAYS** 在任何涉及 issue 的操作前，先判断代码托管平台。

执行 `git remote get-url origin` 获取远程 URL：

| 平台 | 判断条件 | CLI |
|------|----------|-----|
| GitHub | URL 包含 `github.com` | `gh` |
| GitLab | URL 包含 `gitlab.com` 或私有域名 | `glab` |

后续所有 CLI 命令使用判断得到的 CLI 工具。

## 流程

### 1. 收集问题描述

询问用户：
- 要解决的问题
- 潜在的解决方案思路

### 2. 探索代码库

验证问题陈述，理解当前代码状态。

### 3. 讨论备选方案

询问用户是否考虑过其他方案，并呈现可能的替代方案。

### 4. 详细访谈实现细节

对实现方案进行逐项访谈，覆盖所有细节。

### 5. 确定范围边界

明确：
- 计划变更的内容
- 计划不变更的内容

### 6. 检查测试覆盖

检查相关代码的测试覆盖情况。若覆盖不足，询问用户的测试计划。

### 7. 拆分为 tiny commits

将实现拆分为 tiny commits 计划。遵循 Martin Fowler 的建议："每个重构步骤尽可能小，确保程序始终处于可工作状态。"

### 8. 创建 Issue

使用已判断的 CLI 创建 issue：

- GitHub: `gh issue create --title "..." --body "..."` 或 `gh issue create --file -` (从 stdin)
- GitLab: `glab issue create --title "..." --description "..."` 或 `glab issue create --file -` (从 stdin)

## 问题描述

开发者面临的问题，从开发者视角描述。

## 解决方案

问题的解决方案，从开发者视角描述。

## 提交计划

详细实现计划。使用自然语言描述，将实现拆分为尽可能小的提交。每个提交保持代码库处于可工作状态。

## 决策记录

实现过程中做出的决策，包括：

- 将构建/修改的模块
- 模块接口变更
- 技术澄清
- 架构决策
- Schema 变更
- API 契约
- 具体交互方式

**DO NOT** 包含具体文件路径或代码片段——这些很快会过时。

## 测试决策

测试相关决策，包括：

- 良好测试的标准（仅测试外部行为，不测试实现细节）
- 需要测试的模块
- 测试先例（代码库中类似的测试模式）

## 范围外

本次重构不涉及的内容。

## 补充说明（可选）

重构的其他补充信息。
