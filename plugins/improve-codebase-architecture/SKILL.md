# 改进代码库架构

像 AI 一样探索代码库，发现架构摩擦，识别可测试性改进机会，提出模块加深重构方案并创建 issue RFC。

**Deep Module**（John Ousterhout，《软件设计哲学》）用小接口隐藏大实现。Deep Module 更易测试、更易被 AI 导航，允许在边界而非内部进行测试。

## 流程

### 1. 探索代码库

使用 Agent 工具（subagent_type=Explore）自然探索代码库。**DO NOT** 遵循僵化启发式——有机探索并记录遇到的摩擦：

- 理解一个概念需要在多个小文件间跳转的位置
- 模块过浅、接口复杂度接近实现复杂度的位置
- 纯函数仅为测试性提取、但真正 bug 隐藏在调用方式中的位置
- 紧耦合模块在接缝处产生集成风险的位置
- 代码库中未测试或难以测试的部分

遇到的摩擦本身就是信号。

### 2. 展示候选

以编号列表展示加深机会。每个候选展示：

- **集群**：涉及的模块/概念
- **耦合原因**：共享类型、调用模式、概念共属
- **依赖类别**：见 [reference/dependency-categories.md](reference/dependency-categories.md) 的四类依赖
- **测试影响**：哪些现有测试会被边界测试替代

**DO NOT** 此时提出接口。询问用户："要探索哪个候选？"

### 3. 用户选择候选

### 4. 定义问题空间

在生成子代理前，为选定的候选编写面向用户的问题空间说明：

- 新接口需要满足的约束
- 需要依赖的依赖项
- 粗略的代码草图使约束具体化——这不是提案，只是约束的具象化

展示给用户后立即进入步骤 5。子代理并行工作时，用户阅读和思考问题空间。

### 5. 设计多个接口

使用 Agent 工具并行生成 3+ 子代理。每个必须产出**差异显著**的接口设计。

为每个子代理提供独立的技术简报（文件路径、耦合细节、依赖类别、隐藏内容）。给每个代理不同的设计约束：

- 代理 1："最小化接口——最多 1-3 个入口点"
- 代理 2："最大化灵活性——支持多种用例和扩展"
- 代理 3："优化最常见调用者——简化默认用例"
- 代理 4（如适用）："围绕 Ports & Adapters 模式设计，处理跨边界依赖"

每个子代理输出：

1. 接口签名（类型、方法、参数）
2. 使用示例，展示调用者如何使用
3. 内部隐藏的复杂度
4. 依赖策略（如何处理依赖——见 [reference/dependency-categories.md](reference/dependency-categories.md)）
5. 权衡

按顺序展示各设计，然后用文字对比。

对比后给出推荐：哪个设计最强及原因。若不同设计的元素可以组合，提出混合方案。保持观点鲜明——用户需要强有力的判断，而非菜单。

### 6. 用户选择接口（或接受推荐）

### 7. 创建 Issue

判断代码托管平台，使用对应 CLI 创建 issue。判断规则见下方"平台判断"。使用 [reference/issue-template.md](reference/issue-template.md) 中的模板。**DO NOT** 创建前请用户审阅——直接创建并分享 URL。

## 平台判断

通过 `git remote get-url origin` 判断代码托管平台，使用对应 CLI：

| 平台 | 判断条件 | CLI |
|------|----------|-----|
| GitHub | URL 包含 `github.com` | `gh` |
| GitLab | URL 包含 `gitlab.com` 或私有 GitLab 域名 | `glab` |
