# 诊断记录格式

## 文件命名

```
triage-records/<YYYYMMDD>-<HHMM>-DIAGNOSIS.md
```

示例：`triage-records/20260321-1530-DIAGNOSIS.md`

## 文件结构

```markdown
# 诊断记录

**问题**: <问题简要描述>
**时间**: <YYYY-MM-DD HH:MM>
**Issue**: <issue URL 或 "未创建">

---

## 问题描述

**实际行为**: <观察到的现象>
**预期行为**: <应有的正确行为>
**复现步骤**:
1. <步骤 1>
2. <步骤 2>
...

---

## 根因分析

**代码路径**: <涉及的模块/接口>
**失败原因**: <为何当前代码无法正常工作>
**问题类型**: 回归缺陷 / 功能缺失 / 设计缺陷

---

## 修复方案

**最小变更**: <修复根因所需的最小改动>
**受影响模块**: <受影响的模块列表>

---

## TDD 修复计划

### 循环 1
- **RED**: <测试描述>
- **GREEN**: <最小变更>

### 循环 2
- **RED**: <测试描述>
- **GREEN**: <最小变更>

...

### 重构
<全部测试通过后的清理工作>

---

## 验收标准

- [ ] <标准 1>
- [ ] <标准 2>
- [ ] 新测试全部通过
- [ ] 现有测试不受影响
```

## 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 问题 | 是 | 问题简要描述 |
| 时间 | 是 | 生成时间戳 |
| GitHub Issue | 是 | 关联的 issue URL |
| 实际行为 | 是 | 观察到的错误现象 |
| 预期行为 | 是 | 正确行为描述 |
| 复现步骤 | 按需 | 复现问题的步骤 |
| 代码路径 | 是 | 涉及的模块/接口（非具体文件） |
| 失败原因 | 是 | 根因分析 |
| 问题类型 | 是 | 回归/缺失/设计缺陷 |
| 最小变更 | 是 | 修复方案概述 |
| 受影响模块 | 是 | 涉及的模块列表 |

## 耐久性原则

诊断记录应避免：

- 具体文件路径（如 `src/auth/login.ts:42`）
- 行号引用
- 实现细节耦合

应使用：

- 模块名称（如 `认证模块`）
- 行为描述（如 `用户登录时验证 token`）
- 契约定义（如 `AuthService.verify() 返回布尔值`）

## 示例

```markdown
# 诊断记录

**问题**: 用户登录后 token 未持久化
**时间**: 2026-03-21 15:30
**Issue**: https://github.com/owner/repo/issues/123

---

## 问题描述

**实际行为**: 用户刷新页面后需要重新登录
**预期行为**: 刷新页面后保持登录状态
**复现步骤**:
1. 用户登录成功
2. 刷新浏览器页面
3. 返回登录页

---

## 根因分析

**代码路径**: 认证模块 → Token 存储服务
**失败原因**: Token 仅存储在内存中，未写入持久化存储
**问题类型**: 功能缺失

---

## 修复方案

**最小变更**: 在 Token 存储服务中添加 localStorage 写入逻辑
**受影响模块**: 认证模块、存储服务

---

## TDD 修复计划

### 循环 1
- **RED**: 测试登录成功后 token 存在于 localStorage
- **GREEN**: 在登录流程中添加 localStorage.setItem 调用

### 循环 2
- **RED**: 测试页面刷新后从 localStorage 恢复 token
- **GREEN**: 在应用初始化时读取 localStorage 并恢复认证状态

### 重构
统一内存缓存和 localStorage 访问为单一接口

---

## 验收标准

- [ ] 登录成功后 token 写入 localStorage
- [ ] 页面刷新后保持登录状态
- [ ] 新测试全部通过
- [ ] 现有测试不受影响
```
