# 编写 PRD

通过结构化流程创建产品需求文档（PRD），最终提交为 issue。

可根据实际情况跳过非必要步骤。

## 平台判断（流程开始时执行）

**ALWAYS** 在任何涉及 issue 的操作前，先判断代码托管平台。

执行 `git remote get-url origin` 获取远程 URL：

| 平台 | 判断条件 | CLI |
|------|----------|-----|
| GitHub | URL 包含 `github.com` | `gh` |
| GitLab | URL 包含 `gitlab.com` 或私有域名 | `glab` |

后续所有 CLI 命令使用判断得到的 CLI 工具。

## 工作流程

### 1. 收集需求

询问用户：

- 要解决的问题是什么？
- 有哪些潜在的解决方案构想？

获取详细描述。

### 2. 探索代码库

使用 Agent 工具（subagent_type=Explore）验证用户断言，理解代码库当前状态。

目标：

- 确认问题背景的准确性
- 了解现有架构和模式
- 识别相关模块和依赖

### 3. 深度访谈

对计划的每个方面深入追问，直至达成共识。

逐一遍历设计树的每个分支，解决决策之间的依赖关系。

### 4. 模块设计

勾勒实现所需的主要模块（新建或修改）。主动寻找**深度模块**机会。

深度模块特征：

- 通过简单、可测试的接口封装大量功能
- 接口稳定、变更频率低

确认以下内容：

- 模块划分是否符合用户预期
- 哪些模块需要编写测试

### 5. 编写并提交 PRD

完成理解后，按下方模板编写 PRD 并提交为 issue。使用已判断的 CLI：

- GitHub: `gh issue create --title "..." --body "..."` 或 `gh issue create --file -` (从 stdin)
- GitLab: `glab issue create --title "..." --description "..."` 或 `glab issue create --file -` (从 stdin)

## PRD 模板

```markdown
## 问题陈述

用户面临的问题，从用户视角描述。

## 解决方案

问题的解决方案，从用户视角描述。

## 用户故事

完整的用户故事列表。格式：

1. 作为 <角色>，我想要 <功能>，以便 <收益>

示例：
1. 作为手机银行客户，我想要查看账户余额，以便更好地规划消费支出

用户故事应覆盖功能的所有方面。

## 实现决策

实现决策列表，包括：

- 需要构建/修改的模块
- 模块接口变更
- 技术澄清
- 架构决策
- 数据结构变更
- API 契约
- 具体交互方式

**DO NOT** 包含具体文件路径或代码片段。这些内容容易过时。

## 测试决策

测试决策列表，包括：

- 好测试的标准（测试外部行为，而非实现细节）
- 哪些模块需要测试
- 代码库中的类似测试作为参考

## 范围外

不在本 PRD 范围内的内容。

## 补充说明

其他相关说明。
```

## 反模式

- **DO NOT** 跳过代码库探索 - 需要验证用户断言
- **DO NOT** 包含具体文件路径或代码片段 - 容易过时
- **DO NOT** 省略用户故事 - 完整性是关键
